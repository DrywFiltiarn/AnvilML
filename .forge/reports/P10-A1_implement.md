## Summary

Implemented a per-worker keepalive watchdog in `ManagedWorker` that sends `Ping{seq}` messages at configurable intervals over the existing IPC channel and force-kills the child process via `tokio::process::Child::kill()` when a matching `Pong{seq}` is not received within the configured timeout. The ping interval and pong timeout are configurable via `ANVILML_PING_INTERVAL_MS` and `ANVILML_PONG_TIMEOUT_MS` environment variables (defaults: 30s and 10s). The keepalive is started automatically in `WorkerPool::spawn_all()` after each worker's `spawn()` returns.

## Deviations from Plan

- The plan's `inject_handles_for_test()` method signature used `tokio::process::ChildStdin/ChildStdout` parameters, but these opaque types cannot be constructed from in-memory duplex pipes (they require a real child process). The actual test does not use `inject_handles_for_test()` — instead it sets status to Idle directly via `set_status()` and spawns a dummy child for the keepalive kill target. This is a necessary deviation because the test's goal is to verify keepalive behavior without a Python process, which is incompatible with needing real `ChildStdin/ChildStdout` handles.
- The plan's approach used `self.child: Mutex<Option<tokio::process::Child>>` but changed it to `Arc<Mutex<Option<tokio::process::Child>>>` so the keepalive task can clone and share access to the child handle. This was a necessary structural change since the original `Mutex` does not implement `Clone`.
