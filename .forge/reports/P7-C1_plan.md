## Objective

Introduce a `[workspace.dependencies]` table in the root `Cargo.toml` as the single authoritative location for all external dependency versions. Query every currently-declared external dependency via `mcp-rust-docs` (or crates.io fallback) to resolve its current stable version, populate `[workspace.dependencies]`, and migrate every per-crate `Cargo.toml` to reference those shared deps via `{ workspace = true }`.

## Scope

### In Scope
- Add `[workspace.dependencies]` table to root `Cargo.toml` with current stable versions of all 20 runtime external dependencies and 6 dev-dependencies shared across multiple crates
- Migrate 10 per-crate `Cargo.toml` files (`backend`, `crates/anvilml-core`, `anvilml-hardware`, `anvilml-registry`, `anvilml-ipc`, `anvilml-worker`, `anvilml-scheduler`, `anvilml-server`, `anvilml-openapi`) to use `{ workspace = true }` for shared deps
- Internal path dependencies (`anvilml-core`, `anvilml-hardware`, etc.) remain unchanged (no `[workspace.dependencies]` entries)
- Crate-specific feature declarations stay at the crate level alongside `workspace = true`
- Single-use dev-dependencies that appear in only one crate may remain inline
- `Cargo.lock` will be regenerated by cargo — commit the updated lock file

### Out of Scope
- No source code changes in any `.rs` files
- No new external dependencies introduced (only existing ones upgraded)
- No CI workflow changes (covered by P7-B1)
- No feature flag changes beyond `{ workspace = true }` migration
- No changes to `rust-toolchain.toml`

## Approach

### Step 0 — Resolve all dependency versions via MCP lookup

Query every external dependency currently declared across all 10 `Cargo.toml` files. For crates not available on docs.rs, fall back to crates.io (cargo search). Record each lookup result in the `## Resolved Dependencies` section below.

**Compatibility gates:** Three dependencies have major version bumps from their current pins:
- `axum`: 0.7 → 0.8.9 is a **major** bump. Pin at last compatible 0.7.x (latest: **0.7.15**). Document under Blockers.
- `thiserror`: 1.x → 2.0.18 is a **major** bump. Pin at last compatible 1.x (latest: **1.0.69**). Document under Blockers.
- `toml`: 0.8 → 1.1.2 is a **major** bump. Pin at last compatible 0.8.x (latest: **0.8.23**). Document under Blockers.

### Step 1 — Populate `[workspace.dependencies]` in root `Cargo.toml`

Create the table with all resolved versions and required features. Shared deps go here; crate-local-only deps stay in their respective files.

### Step 2 — Migrate per-crate `Cargo.toml` files

For each of the 10 crates, replace inline version strings with `{ workspace = true }`. Preserve:
- All internal path dependencies (`{ path = "..." }`) unchanged
- Crate-specific features declared alongside `workspace = true` (e.g. `tokio = { workspace = true, features = ["macros", "rt-multi-thread", "sync", "time"] }`)
- Target-platform conditional dependencies remain at crate level (e.g. `winapi`, `libloading`)

### Step 3 — Build verification

Run `cargo build --workspace --features mock-hardware`. This will regenerate `Cargo.lock` with the new versions. Fix any compilation errors that arise from version changes.

### Step 4 — Test verification

Run `cargo test --workspace --features mock-hardware`. All tests must pass. Fix any regressions caused by dependency upgrades.

## CI Impact

No CI workflow files are modified in this task (P7-B1 handles CI changes). However, the updated `Cargo.lock` must be committed. CI jobs (`rust`, `rust-windows`) that run `cargo test --workspace --features mock-hardware` will use the new dependency versions — if any version is incompatible with existing code, the build will fail and the task will be blocked (see Risks).

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| `axum 0.7.x` upgrade may introduce subtle API changes (e.g. handler signatures, extractors) | Pin at last compatible 0.7.x; verify with full build + test suite; if compilation fails, fix minimal code changes and document under Deviations |
| `thiserror 1.x` latest may have minor breaking changes in derive macro behavior | Pin at last compatible 1.x (1.0.69); verify compile; fix any derive macro changes minimally |
| `toml 0.8.x` upgrade may change parsing API or serialization behavior | Pin at last compatible 0.8.x (0.8.23); verify compile; check if any runtime usage of toml API changed |
| `tower 0.5.x` is incompatible with axum 0.7.x (axum 0.7 requires tower 0.4) | Keep `tower` pinned at 0.4.x in workspace.dependencies rather than upgrading to latest 0.5.3 |
| `tracing-subscriber` or `futures-util` or `tokio-tungstenite` not available on docs.rs | Fall back to crates.io via cargo search; record the resolved version and note the fallback |
| Dependency upgrade causes transitive dependency conflicts (e.g. multiple versions of same crate) | Let cargo resolve; if conflicts arise, add version constraints in workspace.dependencies to force a single version |
