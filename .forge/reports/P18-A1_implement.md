## Summary

Implemented two lifecycle methods on `WorkerPool` (`restart` and `shutdown_all`) and a
public `ManagedWorker::restart` method to support graceful worker restart and shutdown.
The `tx` field was refactored to use `std::sync::Mutex` with `block_in_place` wrappers
to safely allow sender replacement during respawn while maintaining `Send` across await
points. Two new tests exercise the restart and shutdown_all paths with mock workers.

## Deviations from Plan

- **`tx` field type**: The plan assumed `tx` could be used directly with `.await`. In
  practice, `tx` needed to be wrapped in `std::sync::Mutex` for interior mutability
  (required by `reset_ipc_tx` called from `&self`). The `Send` constraint across await
  points was satisfied by cloning the sender before each `.await` (dropping the
  `MutexGuard` first). This required adding a `tx_lock()` helper that uses
  `tokio::task::block_in_place` to safely acquire the lock from async contexts.
- **`reset_ipc_tx` signature**: Changed from `fn reset_ipc_tx(&mut self)` to
  `async fn reset_ipc_tx(&self)` to work with the `Arc<ManagedWorker>` access pattern
  and the `&self` receiver of `restart`.
- **`start_keepalive` tx capture**: Uses `self.tx.lock().unwrap().clone()` (synchronous
  lock) because the method is not async. This is safe because the cloned `Sender` is
  moved into the spawned task.
- **Pre-existing test fixes**: Fixed `respawn_after_death` test to use `multi_thread`
  runtime (required by `block_in_place` in `reset_ipc_tx`), removed unnecessary `mut`,
  added `.await` to `reset_ipc_tx()` call, and prefixed unused variables with `_`.
