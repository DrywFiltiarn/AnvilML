## Summary

Created `crates/anvilml-worker/src/spawn.rs` implementing `build_command()` and `spawn_worker()` — the subprocess construction layer for Python worker processes. The module constructs a `tokio::process::Command` targeting the platform-specific Python interpreter in the worker venv, with environment variables injected from `WorkerEnv::build()`, stdout/stderr piped for supervisor reading. Added tokio (process feature) and tracing dependencies, re-exported both functions from `lib.rs`, and wrote 5 integration tests in `crates/anvilml-worker/tests/spawn_tests.rs`. Bumped `anvilml-worker` patch version from 0.1.0 to 0.1.1.

## Deviations from Plan

- Added `tracing = "0.1"` as an additional dependency beyond the plan's tokio-only scope. This was required because `spawn_worker()` uses `#[tracing::instrument]` and `tracing::debug!()` — the tracing crate was not listed in the plan's dependencies but is a standard dependency across all other workspace crates (anvilml-artifacts, anvilml-hardware, anvilml-ipc, anvilml-registry, backend).
- Re-exported `build_command` from `lib.rs` in addition to `spawn_worker`. The plan only mentioned re-exporting `spawn_worker`, but the integration tests import `build_command` directly via `use anvilml_worker::build_command;`, requiring it to be public at the crate root.
- The `test_interpreter_path_windows` test is cfg-gated for Windows only and does not inspect internal Command state (same limitation as Unix). This is a known limitation of tokio::process::Command — it wraps std::process::Command which does not expose its internal program path for inspection without spawning.
