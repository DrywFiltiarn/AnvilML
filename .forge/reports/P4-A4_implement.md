## Summary

Created `crates/anvilml-hardware/src/device_db.rs` containing a PCI-ID capability table (`DEVICE_DB`) with 15 curated GPU entries spanning NVIDIA (Ampere, Ada, Hopper), AMD (RDNA2, RDNA3), and Intel Arc architectures. Implemented `resolve_caps_from_row()` which performs a linear scan of the const table to match device PCI IDs and populate `arch`, `caps.fp8`, `caps.flash_attention`, `name`, and `capabilities_source` on a `GpuDevice`. Modified `lib.rs` to add the module declaration and re-exports. Created 7 integration tests in `tests/device_db_tests.rs` covering known NVIDIA, known AMD, unknown device, CPU fallback, VRAM preservation, name overwrite, and table size sanity. Updated `docs/TESTS.md` with entries for all 7 new tests.

## Deviations from Plan

- **DeviceRow struct fields**: The plan specified `DeviceRow` with fields `{ name: &str, arch: &str, fp8: bool, flash_attention: bool }` — without PCI ID fields. However, the plan's approach requires `resolve_caps_from_row` to "Match `dev.pci_vendor_id` and `dev.pci_device_id` against `DEVICE_DB` entries to find the matching row." This is impossible without PCI ID fields in the row struct. Added `pci_vendor_id: u16` and `pci_device_id: u16` fields to `DeviceRow` to make the PCI-ID lookup functional. This is a necessary structural deviation — the plan's API surface was infeasible as written.
- **DEVICE_DB re-export**: The plan listed re-exports of `DeviceRow` and `resolve_caps_from_row` only. Added `DEVICE_DB` to the re-exports because the integration test needs direct access to the constant for the `test_device_db_non_empty` test.
- **Cargo.toml version**: The plan specified bumping from `0.1.3` to `0.1.4`. The version was already `0.1.4` (bumped by prior task P4-A3), so no bump was performed.
- **lib.rs duplicate doc comments**: The original `lib.rs` had two near-identical `//!` doc comment blocks (lines 1-11 and 13-23). Fixed by removing the duplicate and keeping the more complete version. This reduced the file from 74 lines to 64 lines.
