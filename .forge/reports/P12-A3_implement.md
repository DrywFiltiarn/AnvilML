## Summary

Implemented `JobScheduler::submit` in a new `crates/anvilml-scheduler/src/scheduler.rs` file. The method validates the job DAG graph via `validate_graph`, persists the job to SQLite as `Queued` via `insert_job`, enqueues it in the in-memory FIFO queue, broadcasts a `WsEvent::JobQueued` event on the channel, notifies the dispatch loop via `Notify`, and returns a `SubmitJobResponse` with the job ID and 1-based queue position. Added four unit tests covering valid submission, invalid graph rejection, broadcast event delivery, and settings round-trip persistence. Bumped crate version from 0.1.6 to 0.1.7. Added `tracing` dependency for structured logging.

## Deviations from Plan

- **Import ordering**: The plan specified `use tracing;` at the top level of scheduler.rs. Clippy flagged `tracing` as unused at the module level (it's used via the `#[tracing::instrument]` attribute macro and `tracing::info!`/`tracing::field::Empty`). The `tracing` crate IS in dependencies because the `#[tracing::instrument]` derive requires it. The import was removed from the top-level module to avoid clippy warnings; the macro attribute handles re-exporting the macros.
- **JobSettings import**: The plan included `JobSettings` in the parent-module imports. It is only used in tests, so it was moved into the `#[cfg(test)]` module's imports to satisfy clippy's unused-import lint.
- **get_job import**: Similarly, `get_job` is only used in tests and was moved into the test module.
- **tokio features**: The plan specified adding `tokio = { workspace = true, features = ["sync"] }` as a separate dev-dependency line. Instead, I added `"sync"` to the existing runtime dependency (which already had no features specified) and kept the dev-dependency unchanged with `["macros", "rt-multi-thread"]`. This is functionally equivalent but avoids duplicating the tokio entry.
