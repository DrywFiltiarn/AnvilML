## Summary

Removed all `#[serial_test::serial]` attributes and the `serial_test` dev-dependency from
the `backend` crate's integration tests, replacing `#[tokio::test]` with
`#[tokio::test(flavor = "multi_thread", worker_threads = 2)]` across all 12 test functions
in 4 test files. Fixed environment-variable isolation in `preflight_check.rs` by wrapping
both `job_submit_rejected_when_preflight_fails` and `job_submit_proceeds_in_mock_mode` in
`temp_env::async_with_vars` scopes, preventing `ANVILML_WORKER_MOCK` from leaking between
tests running in parallel on the multi-thread runtime.

## Deviations from Plan

- **`preflight_check.rs` — `job_submit_rejected_when_preflight_fails`**: The plan specified
  wrapping the test body in `temp_env::async_with_vars([("ANVILML_WORKER_MOCK", None::<&str>)],
  async { ... })`. During testing, this alone was insufficient to prevent `ANVILML_WORKER_MOCK`
  from leaking from the parallel `job_submit_proceeds_in_mock_mode` test (which sets it to "1"
  at the process level). Added `std::env::remove_var("ANVILML_WORKER_MOCK")` inside the
  `temp_env` scope as a defense-in-depth measure, and also added
  `("ANVILML_MOCK_DEVICE_TYPE", Some("cpu"))` and `("ANVILML_MOCK_VRAM_MIB", Some("0"))` to
  the scope to ensure a clean CPU mock context.

- **`preflight_check.rs` — `job_submit_proceeds_in_mock_mode`**: The plan did not mention
  changing this test, but since `#[serial]` was removed and the test sets
  `ANVILML_WORKER_MOCK=1` at the process level, it was necessary to wrap it in
  `temp_env::async_with_vars([("ANVILML_WORKER_MOCK", Some("1"))], async { ... })` to
  prevent the variable from leaking to `job_submit_rejected_when_preflight_fails`. Replaced
  the manual save/restore pattern (`std::env::set_var` + conditional `remove_var`) with
  the `temp_env` scope.
