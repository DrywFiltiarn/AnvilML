## Summary

Fixed a cross-platform bug in `crates/anvilml-worker/src/managed.rs` where `GenericFilePath` was used to convert IPC socket paths to `interprocess` names on all platforms. On Windows, `build_socket_path()` returns a named pipe path (`\\.\pipe\anvilml-worker-...`) which must be converted via `NamedPipe` (the Windows local socket name type) instead of `GenericFilePath`. Added a private `to_socket_name()` helper with `#[cfg(unix)]` and `#[cfg(windows)]` arms, replaced all 6 call sites of `.to_fs_name::<GenericFilePath>()` on `build_socket_path()` results with `to_socket_name(&path)`, and updated module-level imports to conditionally include `GenericFilePath` (unix+test) and `NamedPipe` (windows only).

## Deviations from Plan

- **API correction**: The plan specified `GenericNamespaced` and `ToNsName` for the Windows code path. However, the interprocess 2.4.2 API does not expose a `to_ns_name` method or a `GenericNamespaced` type that implements `PathNameType`. The correct Windows path type is `NamedPipe` (from `interprocess::os::windows::local_socket::NamedPipe`), used with the existing `ToFsName::to_fs_name` method. This was discovered during the Windows cross-compilation check and corrected before proceeding.
- **Import gating**: `GenericFilePath` was gated behind `#[cfg(any(unix, test))]` instead of being in the unconditional import list, because it is only used in `#[cfg(unix)]` code (the `to_socket_name` helper's Unix arm) and in `#[cfg(test)]` code. On the Windows cross-compile target (non-test), it would be unused.
