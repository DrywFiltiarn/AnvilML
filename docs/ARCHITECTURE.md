## 1. What AnvilML Is

AnvilML (`anvilml` / `anvilml.exe`) is the Rust backend binary of the SindriStudio
image-generation platform. It:

- Spawns and supervises one Python worker process per GPU (or one CPU worker as fallback).
- Exposes a versioned REST + WebSocket API as the **sole integration surface** for any client.
- Manages job scheduling, model registry, artifact storage, and all state via SQLite.
- Is **headless only** — it does not serve a web UI, does not host static files, and does
  not act as a reverse proxy. It is a pure API server.
- Runs the real inference code path always — there is no compiled-out or
  permanently-mocked real path anywhere in this codebase (`ANVILML_DESIGN.md §14.1`).
  Mock mode is a second, equally-maintained branch, never a substitute for an unbuilt one.

**AnvilML is not SindriStudio.** SindriStudio is the separate launcher that starts
AnvilML and BloomeryUI as independent sibling OS processes. AnvilML never references
or embeds BloomeryUI. These are permanently separate binaries.

**MVP model scope:** Z-Image Turbo (ZiT) + Qwen3 4B; Flux 2 Klein 4B + Qwen3 4B;
Flux 2 Klein 9B + Qwen3 8B. Full matrix: `ANVILML_DESIGN.md` Appendix B.

## 3. Crate Dependency Graph

```
anvilml-core  (no deps — pure data, zero I/O, zero async)
  ├── anvilml-hardware      (← core)
  ├── anvilml-registry      (← core)
  ├── anvilml-artifacts     (← core)
  └── anvilml-ipc           (← core)
        └── anvilml-worker  (← ipc, hardware, core)
              └── anvilml-scheduler  (← worker, registry, artifacts, core)
                    └── anvilml-server  (← all above)
                          └── backend/src/main.rs

anvilml-openapi  (← core, server — build-time only; not in release binary)
```

**Hard rule:** no crate may depend on a crate above it in this graph. No cycles.
Any new dependency must maintain this order. Violations are a compilation error.

**This exact graph was the source of the single most serious design-process failure
recorded in this project's history** — an agent relocating a module across this
boundary on the basis of a claimed circular-dependency problem that did not actually
exist. Before accepting any claim of the form "X must depend on Y because otherwise
there is a cycle," check it against this diagram and the real `Cargo.toml` files
first. Full detail: `ANVILML_DESIGN.md §3.2`.

## 4. Crate Responsibilities

| Crate | Owns | Must not do |
|:------|:-----|:------------|
| `anvilml-core` | All domain types; config schema; `AnvilError` enum | I/O, async, network, subprocess |
| `anvilml-hardware` | GPU/CPU enumeration via Vulkan (primary) + platform fallbacks; VRAM refresh | Panic on missing driver — always return at least one CPU device. Claim to know real torch-level compute capability — that is the Python worker's job (§7 above; `ANVILML_DESIGN.md §6.1`) |
| `anvilml-registry` | Scan model dirs; persist `ModelMeta` to SQLite; SHA256-gated SQL seed runner; the PCI-ID capability hint table (`DeviceCapabilityStore`) | Cache model file contents in memory |
| `anvilml-artifacts` | Content-addressed PNG artifact storage; persist `ArtifactMeta` to SQLite; shared independently by scheduler and server | Know about HTTP request/response types; know about IPC message framing |
| `anvilml-ipc` | ZeroMQ ROUTER socket wrapper (split send/recv halves, §6 below); `WorkerMessage` / `WorkerEvent` enums; msgpack serialisation; `EventBroadcaster` (placed here to avoid an `anvilml-worker`/`anvilml-server` cycle) | Process management; persistence; business logic beyond event relaying; know about `ManagedWorker` or any worker lifecycle state |
| `anvilml-worker` | Spawn/supervise/respawn Python worker subprocesses (real-mode by default — §9); IPC bridge; keepalive heartbeat; demux register/deregister pair | Contain logic beyond process management and message routing |
| `anvilml-scheduler` | Job queue; VRAM ledger; DAG validation using dynamic registry; dispatch loop; FP8-capability gate at dispatch time | Know about HTTP request/response types |
| `anvilml-server` | axum router; all HTTP handlers | Contain business logic — handlers call into scheduler/worker/registry/artifacts only |

## 5. Cargo Feature Flags

| Feature | Declared in | Forwarded by | Effect |
|:--------|:------------|:-------------|:-------|
| `mock-hardware` | `anvilml-hardware` | `anvilml-worker`, `anvilml-scheduler`, `anvilml-server`, `backend` | Replaces GPU detection with `MockDetector` driven by `ANVILML_MOCK_*` env vars. **All CI builds use this flag.** Never enabled in release builds. (`ANVILML_FORCE_WORKER_MOCK=1` triggers the same effect on a non-`mock-hardware` binary — see `ENVIRONMENT.md §3.5`.) |

**Forwarding rule** — every crate that depends (directly or transitively) on
`anvilml-hardware` must declare:
```toml
[features]
mock-hardware = ["anvilml-hardware/mock-hardware"]
```
The Python-side mock (`ANVILML_WORKER_MOCK=1`) is a separate, orthogonal mechanism
that does not require recompilation, and is never used to substitute for a missing
real code path — see §9.

## 6. IPC Protocol Summary

See `ANVILML_DESIGN.md §8` for the full specification, including the ownership
model in §8.3 — read that before touching `anvilml-ipc` or `anvilml-worker`.

**Transport:** ZeroMQ ROUTER (Rust supervisor) ↔ DEALER (Python worker) over
`tcp://127.0.0.1:{port}`. The supervisor binds one ROUTER socket for the entire server
process before spawning any workers. Each worker connects a DEALER socket with its
`worker_id` as the ZeroMQ identity. Routing is handled automatically by ZeroMQ.

**Port:** OS-assigned at bind time. Passed to workers via `ANVILML_IPC_PORT` env var.

**Serialisation:** msgpack via `rmp-serde` (Rust) / `msgpack` (Python). All messages are
flat dicts with a `_type` discriminator key matching the enum variant name.

**Message directions:**
- `WorkerMessage` (Rust → Python): `Ping`, `Shutdown`, `Execute`, `CancelJob`, `MemoryQuery`
- `WorkerEvent` (Python → Rust): `Ready`, `Pong`, `Dying`, `MemoryReport`, `Progress`, `ImageReady`, `Completed`, `Failed`, `Cancelled`

The `Ready` event carries the complete list of `NodeTypeDescriptor` objects registered
by the Python worker, plus `capabilities_source` (`"pytorch"` in real mode, `"mock"`
in mock mode — never absent or ambiguous). The Rust scheduler stores the node types
in `NodeTypeRegistry` — there is no compile-time node type list.

**`RouterTransport`'s send and receive halves are split at construction**, not
guarded by one shared lock. This is the fix for a real, recorded shutdown deadlock
(a blocked `recv()` held the same lock a concurrent `Shutdown` send needed). See
`ANVILML_DESIGN.md §8.3` before changing `transport.rs`.

## 7. Device Capability Hint Table

`docs/SUPPORTED_DEVICES_DB.md` was the one-time, hand-authored source for the PCI-ID
→ capability rows in `database/seeds/devices.sql`. That conversion already happened;
it is not a build step and has no automated equivalent in this codebase. The Markdown
file stays in `docs/` permanently as a human reference and **must never be deleted by
any task**, but no code reads it. Any future change to device capability data is made
directly to `devices.sql` (or by an external process outside this repo), never by
editing the Markdown and expecting a regeneration step — none exists, by design.
Full detail and the reasoning for removing the old `device_db.rs` Rust table in
favour of this single SQLite-backed source: `ANVILML_DESIGN.md §7.5`.

## 8. Generic Node System Summary

See `ANVILML_DESIGN.md §10` for the full specification, including the §10.4 method
naming contract and the §10.6 mock/real parity marker requirement — both apply to
every node and arch module, with no exceptions.

**There are no architecture-specific node families.** There is no `ZitSampler`,
`Flux2KleinTextEncode`, or any architecture-prefixed node type. The baseline MVP
nodes are:

| Node | Category | Key inputs → outputs |
|:-----|:---------|:---------------------|
| `LoadModel` | Loaders | `model_id` → `MODEL` |
| `LoadVae` | Loaders | `model_id` → `VAE` |
| `LoadClip` | Loaders | `model_id`, `clip_type?` → `CLIP` |
| `ClipTextEncode` | Conditioning | `CLIP`, `positive_text`, `negative_text?` → `CONDITIONING` |
| `EmptyLatent` | Latents | `width`, `height`, `model?` → `LATENT` |
| `Sampler` | Sampling | `MODEL`, `CONDITIONING`, `CLIP`, `LATENT`, `steps`, `cfg`, `seed` → `LATENT`, `seed` |
| `VaeDecode` | Decoding | `VAE`, `LATENT` → `IMAGE` |
| `ImageResize` | Images | `IMAGE`, `width`, `height` → `IMAGE` |
| `SaveImage` | Output | `IMAGE`, `seed?`, `steps?` → *(none)* |

`LoadModel` outputs only `MODEL`. The VAE is always an explicit `LoadVae → VAE` path wired
into `VaeDecode`. There is no implicit VAE propagation from any other node.

Architecture-specific dispatch lives in `worker/nodes/arch/{diffusion,clip,vae}/`
modules. Every arch module, regardless of family, exposes the same fixed method
names — `can_handle()`, `get_module()` (shared dispatcher), `load()`, and (where
applicable) `sample()` / `decode()` / `compute_latent_shape()`. **Never**
`load_model()`, `load_vae()`, `load_clip()`, or any other family-prefixed variant —
this exact inconsistency existed across arch modules in a prior design and is closed
permanently by the fixed-name table in `ANVILML_DESIGN.md §10.4`.

## 9. Worker Lifecycle & Real-Path Startup Summary

See `ANVILML_DESIGN.md §9` (Rust-side lifecycle, the `WorkerHandle`/`ManagedWorker`
ownership split) and §14 (Python-side startup) for full detail.

**`ManagedWorker::run()` owns its own lifecycle task end-to-end** — there is no
separate by-value `shutdown()` method competing with it for ownership of `self`.
`WorkerPool` holds a `Vec<WorkerHandle>` (a cheap, `Clone`-able struct sharing a
status lock and a shutdown-trigger channel), never `Arc<ManagedWorker>` directly.
This shape exists specifically because the earlier shape (`Arc`-wrapping a struct
with by-value `run(self)`/`shutdown(self)` methods) made `run()` impossible to call
once wrapped — a real, recorded regression. Re-deriving worker ownership from
scratch is not necessary; use the shape in `ANVILML_DESIGN.md §9.1` directly.

**`worker_main.py` has no mock-only startup gate.** Real-mode startup (import torch,
select device, run the real `capability.probe_capabilities()`, register nodes, send
`Ready` with `capabilities_source: "pytorch"`) is the default branch every time
`ANVILML_WORKER_MOCK` is unset. Mock mode is a second, equally-maintained branch
(synthetic capability values, never imports torch) — never a stand-in for an
unwritten real branch. The two sequences differ in exactly one step (capability
probing); IPC connection, node import, and the dispatch loop are identical code in
both modes. Full sequences: `ANVILML_DESIGN.md §14.2`–§14.3, also in
`ENVIRONMENT.md §5`.

**Every node and arch-module function ships both a mock-mode and a real-mode test**,
cross-referenced by a `REAL_PATH_VERIFIED`/`MOCK_PATH_VERIFIED` marker pair next to
the function (`ANVILML_DESIGN.md §10.6`). Real-mode tests use tiny synthetic fixture
checkpoints under `worker/tests/fixtures/`, never real downloaded model weights and
never the production-size checkpoint — this is what makes real-mode testing
practical on the 10GB-RAM Forge agent VM and in GitHub CI.

## 11. File Size Guidelines

These thresholds trigger a **mandatory justification**, not an automatic split. When a
file reaches its threshold the agent must stop and ask: *does this file own one coherent
concern, or has it grown because unrelated logic was added?* If the answer is the latter,
split. If the file genuinely owns a single concern that requires this much code — a complex
state machine, a large protocol decoder, a comprehensive test suite for one module — keep
it whole and note the justification in `## Deviations from Plan`.

Splitting purely to hit a line count produces worse architecture than a coherent large
file. The goal is cohesion, not brevity.

| File type | Review threshold | Common split signal |
|:----------|:----------------|:--------------------|
| Rust source (`.rs`) | 400 lines | File mixes data types, business logic, and utilities |
| Python source (`.py`) | 350 lines | File mixes I/O, computation, and configuration |
| Test files (any language) | 500 lines | Tests for more than one logical unit of behaviour |
| `lib.rs` in any crate | 80 lines | Contains implementation code (never appropriate) |

**`lib.rs` is the one absolute rule**: it must contain only `pub mod`, `pub use`, and the
crate-level `//!` doc comment. Implementation code in `lib.rs` is never appropriate
regardless of how small it is; the 80-line threshold exists because legitimate `lib.rs`
content never approaches it.

**`#[cfg(test)]` inline blocks** in Rust source files are discouraged except for trivial
unit tests of a single pure function (≤ 20 lines, no test helpers, no I/O). Prefer
`crates/{name}/tests/` for all but the most trivial cases. Python tests always go in
`worker/tests/`.

## 12. CI Gates Summary

See `docs/ENVIRONMENT.md §6–8` for full commands.

**GitHub CI jobs** (run on every push to `main`):

| Job | Runner | Purpose |
|:----|:-------|:--------|
| `rust-linux` | ubuntu-latest | `cargo fmt --all -- --check`, clippy, full Rust test suite (`--features mock-hardware`) |
| `rust-windows` | windows-latest | clippy, full Rust test suite (`--features mock-hardware`) — no `fmt --check` |
| `worker-linux-mock` | ubuntu-latest | Provision venv (no torch); `ANVILML_WORKER_MOCK=1 python -m pytest worker/tests -v -m "not real_mode"` |
| `worker-linux-real` | ubuntu-latest | Provision venv + torch CPU wheel; `python -m pytest worker/tests -v -m real_mode` (mock unset) |
| `worker-windows-mock` | windows-latest | Same as `worker-linux-mock`, Windows paths |
| `worker-windows-real` | windows-latest | Same as `worker-linux-real`, Windows paths |
| `openapi-drift` | ubuntu-latest | Regenerate `openapi.json`, assert no diff |
| `config-drift` | ubuntu-latest | `cargo test -p anvilml --features mock-hardware -- config_reference` |

The mock/real `worker-*` job split (new in v4) exists so a real-mode CPU failure is
never masked by an unrelated mock-mode failure's exit code in the same job.

**Local WSL2 pre-push gate** (not a CI job — run before every push):
```bash
cargo check --workspace --features mock-hardware --target x86_64-pc-windows-gnu
```
This gate catches Windows-incompatible code without requiring a Windows runner on every push.
The `x86_64-pc-windows-gnu` target must be installed: `rustup target add x86_64-pc-windows-gnu`.

**Local-only, every ACT session (not a CI job):** mandatory build cache cleanup
(`cargo clean` + Python cache removal) after every session that ran a build/test
command — `ENVIRONMENT.md §13`. Does not apply to CI; ephemeral runners have nothing
to clean.
